# Local Cloud Sandbox (Terraform + LocalStack + Docker)

This project demonstrates the deployment of a local cloud environment (AWS Sandbox) using the **Infrastructure as Code (IaC)** approach. The architecture emulates core AWS services inside Docker containers, eliminating the need to incur costs in the real AWS cloud during development and testing processes.

## 🛠️ Tech Stack
* **Terraform (IaC)** – defining, versioning, and automating infrastructure resources.
* **LocalStack (v3.4.0)** – local AWS cloud emulator (EC2, S3) running inside a container.
* **Docker & Docker Compose** – containerization and orchestration of the cloud environment.
* **Linux (Ubuntu Server)** – host operating system for the environment.

## 📐 Project Architecture
Using Terraform code, the following resources were automatically provisioned inside the LocalStack environment:
1. **Virtual Server (AWS EC2 Instance)** – a `t2.micro` instance tagged as a portfolio server.
2. **Object Storage (AWS S3 Bucket)** – a bucket named `portfolio-project-bucket` dedicated to storing static objects.

## 🧠 Challenges & Troubleshooting (What I Learned)

This project was an excellent opportunity to practice advanced debugging of system, network, and container issues that DevOps engineers face in daily production work:

### 1. Storage Management & LVM Volumes in Linux
* **Issue:** During provider download, Terraform threw a critical `no space left on device` error due to virtual machine disk constraints.
* **Solution:** I analyzed disk utilization using `df -h` and cleared the Docker cache (`docker system prune`). Then, after physically expanding the virtual machine disk in the hypervisor, I safely extended the main system partition managed by the **LVM** mechanism using the commands `lvextend -l +100%FREE /dev/mapper/ubuntu--vg-ubuntu--lv` and `resize2fs`. This provided the system with the necessary space for stable operation.

### 2. The Danger of the `:latest` Tag & Version Pinning
* **Issue:** The LocalStack container unexpectedly terminated shortly after startup with `exit code 55 (License activation failed)`.
* **Solution:** Container log analysis (`docker compose logs`) revealed that the default image with the `:latest` tag pulled a development build, which automatically enforced paid Pro features. I applied a DevOps best practice – changed the configuration to a specific, stable community version tagged `3.4.0` (Community Edition), which permanently resolved the issue.

### 3. Network Routing & S3 Path Style
* **Problem:** The `terraform apply` command would hang indefinitely during the `Refreshing state...` phase while attempting to reach LocalStack.
* **Solution:** I diagnosed that the local DNS inside the virtual machine could not properly resolve subdomain patterns like `bucket.s3.localhost.localstack.cloud` generated by default by the AWS provider. I configured communication directly via the IP address `127.0.0.1` and forced the `s3_use_path_style = true` flag in the Terraform code. This forced Terraform to query LocalStack directly via URL paths (e.g., `127.0.0.1:4566/bucket`), immediately unblocking the communication pipeline.

## 🚀 How to Run the Project Locally

1. Clone the repository and navigate to the project folder:
   
   ```bash
   cd local-cloud-sandbox
   ```
   
Start the local AWS emulator in Docker:
   ```bash
   sudo docker compose up -d
   ```
Initialize Terraform (download plugins and providers):
   ```bash

    terraform init
   ```
Check the infrastructure deployment plan:
   ```bash

    terraform plan
   ```
Deploy the infrastructure to the local cloud:
   ```bash
    terraform apply
   ```

 

After testing, destroy all provisioned resources:
   ```bash

    terraform destroy

   ```
